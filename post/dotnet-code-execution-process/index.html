.NET Framework Code Execution Process

24 February 2022

In a previous post I wrote about Managed Code vs Unmanaged Code, and that got me thinking… when you write, compile, and run an application, there’s a lot of things going on under the hood. I think it’s important to have an idea of the process so you can better understand how your applications work. Since I mostly work in the .NET world myself, I’m going to briefly explain the typical process that occurs when managed .NET code is compiled and executed by the .NET Framework.

One of the main advantages of the .NET Framework is that it allows you to use different .NET programming languages (C#, VB.NET, Visual C++, F#) interactively. It also allows you to run .NET code on different operating systems and CPUs.

Common Intermediate Language (CIL)

When a .NET application is compiled, the language compiler compiles the .NET source code into Common Intermediate Language (CIL) first, rather than CPU specific machine code. An assembly file is generated containing the CIL code. The assembly file will have a .dll extension when compiled from a class library or web project or it will have a .exe extension when compiled from a console or desktop application.

It’s the CIL code and Common Language Runtime that allows the various .NET languages to be used interactively and run on different operating systems and CPUs.

Common Language Runtime (CLR)

.NET programs are executed from the Common Language Runtime (CLR) virtual machine. The CLR manages the execution of .NET programs and provides services like memory management, type safety, garbage collection, etc.

Just-in-time (JIT)

The CLR includes a just-in-time (JIT) compiler, which compiles code at run time. The JIT compiler in the CLR converts the CIL code into native machine code that can be executed by the CPU.
